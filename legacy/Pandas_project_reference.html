About this reference — how it is structured and why

An analysis does not begin with a file. It begins with a question for which numbers might form part of the answer — and a file is one of many intermediate states between that question and the eventual response. The «messy CSV» is not the starting point; it is the point at which the data first becomes visible and uncomfortable.

Before reaching for any command, the analyst should be able to state three things in plain language: (i) what question is being asked, (ii) which population or unit the answer concerns, and (iii) what an answer would look like if one were given the result in finished form. Only then is it productive to ask which tools are required.

The remainder of this reference is organised around the typical journey from such a question to a delivered result. Each stage lists the commands one is likely to encounter, with — for each command — a description of what it does, a token-by-token reading of what the code itself means, the conditions under which it applies, and the situations in which one would reach for it.

1 Locate & Load — bringing the data into Python

Loading is the operation by which a file on disk is read into a Python object that can be queried, transformed and summarised. The choice of loader depends on the file format; the choice of parameters depends on what the file's author did with characters, separators, and missing markers.
How to choose: The file extension is a useful first signal, but not authoritative. Open the file in a plain-text editor first and read the first ten lines. Establish: which character separates fields, whether the first row contains column names, how the file encodes special characters (German umlauts are a frequent indicator), and how missing values are written. Only then call a loader.

2 Inspect — understanding what was loaded

Before any analytical operation, the analyst should be able to describe the loaded object in four respects: its size, the type of each column, the prevalence of missing values, and the presence of duplicate records. These four checks correspond to questions a careful reader would ask before trusting any subsequent claim.
How to choose: Begin with shape and head(); these confirm that the loader produced what was expected. Follow with info() and isna().sum(); these reveal the structural integrity of the table. Use value_counts() and describe() when interest shifts to the distribution of a single column.

3 Clean — addressing irregularities

Cleaning denotes the corrections applied so that the data can be analysed without introducing systematic error. The most common irregularities are inconsistent types, missing values, duplicate records, and inconsistent string representations of the same underlying category. Each requires a deliberate decision; defaults rarely capture clinical intent.
How to choose: Cleaning proceeds in a fixed order: types are corrected first (so that subsequent comparisons work), then missing values (so that they do not propagate silently), then duplicates (so that aggregations are not inflated), then string normalisation (so that categories collapse correctly). Decisions about how to treat missingness are addressed in the Decision Boxes at the end of this document.

4 Transform — deriving variables, filtering, sorting

Transformation reshapes the cleaned table toward the question. New variables are computed from existing ones; rows are filtered to the subpopulation of interest; the order of the table is chosen so that a reader can locate the relevant rows quickly.
How to choose: If a new value is to be computed for every row, use a vectorised expression (column arithmetic, np.where, np.select). If only a subset of rows is of interest, build a boolean mask. If a top-N or bottom-N is required, prefer nlargest / nsmallest over a full sort.

5 Aggregate — summarising by group

Aggregation reduces a long table to a short one by computing a summary statistic within each group. The mental model is split–apply–combine: split the table by one or more key columns, apply a function to each group, combine the results into a single summary.
How to choose: groupby followed by an aggregation produces a long-format summary suitable for further analysis. pivot_table produces a wide-format summary suitable for display. transform is appropriate when the group-level statistic must be aligned back to every original row (for instance, to compute a deviation from group mean).

6 Merge — combining tables and arrays

Most analyses require information from more than one source. Combining sources may take three forms: joining tables by a shared key, stacking tables that share structure, or complementing one source with another to fill in missing values. NumPy arrays are combined in analogous, but simpler, ways.
How to choose: If the two tables share a key column and each row of the first should pick up additional columns from the second, the operation is a merge. If the two tables have the same columns and represent different periods or sources, the operation is a concatenation. If the two tables describe the same entities but each contains gaps the other can fill, the operation is a complement (combine_first).

7 Visualize — graphical summaries

A graphical summary supports the reader in seeing a pattern that a tabular summary cannot. The choice of plot type follows from the question: comparison across categories (bar), evolution over time (line), distribution of a single variable (histogram), relation between two variables (scatter), distribution by group (boxplot).
How to choose: Matplotlib's object-oriented interface (fig, ax = plt.subplots() followed by methods on ax) is the recommended foundation. Seaborn provides briefer syntax for common statistical plots and is appropriate once the foundations are familiar. Every plot intended for an external reader requires, at minimum, a descriptive title, axis labels with units, and — if multiple series are present — a legend.

8 Export — delivering the result

A result that remains in a notebook is not yet delivered. Delivery takes the form of data files for downstream use, image files for inclusion in reports, and the notebook itself rendered for an external reader.
How to choose: Tabular data intended for re-use by humans should be exported as CSV or Excel. Tabular data intended for re-use by other Python programs is better stored as Parquet. Figures intended for slides should be exported as PNG at high resolution; figures intended for printed reports as PDF or SVG.

★ Decision boxes — the considered choices

Three decisions recur in any non-trivial analysis. Each admits several defensible answers; the correct answer depends on the question and on the structure of the missingness or the join.

Missing values — drop rows, drop columns, fill, or allow to propagate?

A column contains NaN. The decision is determined by two considerations: why the value is missing, and how essential the column is to the question.

Drop the row — dropna(subset=[col]) Indicated when the column is essential to the analysis (e.g. patient_id, admission_date) and no substitute is defensible. Acceptable if fewer than approximately five per cent of rows are lost.
Drop the column — drop(columns=[col]) Indicated when the column is more than approximately half missing and not load-bearing for the question, or is redundant with another column. The decision should be recorded explicitly in the notebook.
Impute — fillna(median) / ffill Indicated when missingness is plausibly random, the column is numeric, and a substitute is defensible. The median is robust to skew; the mean only for symmetric distributions; forward-fill for time series whose state changes infrequently.
Allow to propagate Indicated when the downstream aggregation is itself robust to NaN (pandas's mean, sum, and groupby skip NaN by default). Only when a NaN-propagating result is required does one specify skipna=False.

The decision should be made per column, not per DataFrame. dropna() applied to the whole DataFrame, with default arguments, is almost always the wrong choice on real data.

Merging two DataFrames — which join, which method?

Enrich table A with columns from Ba.merge(b, on=key, how="left"). Retains every row of A; adds matching columns from B, with NaN where no match exists.
Restrict to rows present in botha.merge(b, on=key, how="inner"). Smaller result; no missing values introduced by the merge.
Union of all keysa.merge(b, on=key, how="outer", indicator=True). Permits auditing of which keys are present in only one source.
A is the reference; B fills the gapsa.combine_first(b). Applied cell by cell: where A has a value, A is retained; where A is NaN, B is used.
Stack vertically (same columns)pd.concat([a, b], ignore_index=True).
Glue side by side (same rows)pd.concat([a, b], axis=1).

NumPy — which array-combination function?

1D + 1D, end to endnp.concatenate([a,b]) or np.hstack([a,b]). Produces [a₀…aₙ, b₀…bₘ].
2D + 2D, stack as further rowsnp.concatenate([A,B], axis=0) or np.vstack([A,B]).
2D + 2D, glue as further columnsnp.concatenate([A,B], axis=1) or np.hstack([A,B]).
Two 1D arrays → 2D with each as a columnnp.column_stack([a,b]). Shape (n, 2).
Two 1D arrays → 2D with each as a rownp.vstack([a,b]). Shape (2, n).
Introduce a new axisnp.stack([a,b]). Used for batching.

Mnemonic: the argument axis denotes the dimension along which the result grows. axis=0 produces additional rows; axis=1 produces additional columns.

▣ A complete worked example

A short, self-contained analysis that traces all eight stages on synthetic data — generated inline so the script runs in any notebook without an external file. Two questions are addressed of the Saint Mary's data: which wards have the longest length of stay, and which DRGs are loss-making.
What the script does, by stage:
(1) generates a small Saint Mary's dataset with deliberate irregularities; (2) inspects shape, types, and missingness; (3) cleans gender labels, sentinel values, and missing length of stay; (4) derives margin and a loss indicator; (5) aggregates length of stay by ward and margin by DRG; (6) merges encounters with the DRG catalogue and the patient register; (7) produces a two-panel figure; (8) writes the summaries and the enriched table to disk.
The script may be copied into a Jupyter notebook and executed top to bottom. It requires pandas, numpy, and matplotlib — already part of a standard scientific Python installation.
# ============================================================
# Saint Mary's — a complete, runnable example
# Question: Which wards have the longest length of stay,
#           and which DRGs are loss-making?
# ============================================================

import numpy as np
import pandas as pd
import matplotlib.pyplot as plt
from pathlib import Path

# --- Step 1: data ----------------------------------------------
# In production one would load files:
#   patients   = pd.read_csv("data/patients.csv")
#   encounters = pd.read_csv("data/encounters.csv")
# Here the tables are generated so the script is self-contained.

# np.random.default_rng creates a *pseudo-random number generator* — an
# object from which subsequent calls draw random values. The 'seed'
# argument fixes its starting state, so that every run produces the same
# sequence of values. This is essential for reproducibility: a colleague
# running the same script obtains the same example data.
rng = np.random.default_rng(seed=42)

patients = pd.DataFrame({
    "patient_id": [f"P{i:03d}" for i in range(60)],
    "age": rng.integers(18, 90, size=60),
    "gender": rng.choice(
        np.array(["M", "F", "m", "female", "männlich", " male  ", None], dtype=object),
        size=60),
})

encounters = pd.DataFrame({
    "encounter_id": [f"E{i:04d}" for i in range(100)],
    "patient_id": rng.choice(patients["patient_id"], size=100),
    "ward": rng.choice(
        np.array(["ICU", "Cardiology", "Orthopaedics", "General", None], dtype=object),
        size=100),
    "drg_code": rng.choice(["A01", "A02", "B05", "C10", "D12"], size=100),
    "los_days": rng.integers(1, 21, size=100).astype(float),
    "cost_eur": rng.normal(2500, 800, size=100).round(2),
})
# Inject realistic irregularities
encounters.loc[rng.choice(100, 5, replace=False), "los_days"] = np.nan
encounters.loc[rng.choice(100, 3, replace=False), "cost_eur"] = -1   # sentinel for "invalid"

drg_catalog = pd.DataFrame({
    "drg_code":          ["A01", "A02", "B05", "C10", "D12"],
    "reimbursement_eur": [3200, 2100, 1800, 4500, 1500],
})

# --- Step 2: inspect -------------------------------------------
print("Shape:", encounters.shape)
print(encounters.dtypes)
print("Missing values per column:")
print(encounters.isna().sum())

# --- Step 3: clean ---------------------------------------------
# 3a — *normalise* the gender labels.
# To normalise a column means to bring its values into a single, agreed
# form. A single underlying category may appear in several spellings
# ("M", "m", "Male", "männlich", with stray whitespace). Before any
# grouping or comparison on 'gender' can be trusted, the strings must be
# collapsed to a canonical form. The .str *accessor* applies string
# methods element-wise to a Series:
#   .strip()    removes leading and trailing whitespace;
#   .lower()    converts every character to lowercase;
#   .replace()  then maps the remaining variants onto two canonical labels.
patients["gender"] = (patients["gender"]
                      .str.strip()
                      .str.lower()
                      .replace({"m": "male", "männlich": "male", "f": "female"}))

# 3b — translate the *sentinel value* into NaN, then impute.
# A sentinel value is a special, agreed-upon number that a source system
# uses to signal "no valid value here" — most often because the export
# file format cannot store NaN directly. Hospital information systems
# frequently use -1, -99, or 9999 for this purpose. In the present data,
# the cost export uses -1 for invalid entries.
#
# Until -1 is converted to NaN, the affected rows look like genuine costs
# of minus one euro. Any subsequent statistic — the mean, the sum, the
# distribution of margins — would be distorted by these pseudo-values.
#
# The two-step operation handles this:
#   .replace({-1: np.nan}) substitutes the sentinel with np.nan, which is
#                          pandas's proper marker for a missing value;
#   .fillna(median)        then *imputes* (i.e. estimates and fills in)
#                          those NaN cells using the column's median —
#                          a robust central value that is unaffected by
#                          outliers (the mean would be pulled by extremes).
encounters["cost_eur"] = (encounters["cost_eur"]
                          .replace({-1: np.nan})
                          .fillna(encounters["cost_eur"].median()))

# 3c — length of stay is essential to the analysis, so rows that lack it
# are dropped. After rows are dropped, the row labels (the *index*) have
# gaps — 0, 1, 3, 4, 7, ... — because the dropped numbers are simply
# absent. .reset_index(drop=True) renumbers the rows to a clean 0...n-1
# sequence; 'drop=True' discards the old index instead of retaining it
# as an additional column.
encounters = encounters.dropna(subset=["los_days"]).reset_index(drop=True)

# 3d — a missing ward is informative in itself: it indicates that the
# source system failed to record where the patient was treated. Rather
# than discarding the row or imputing a fictitious ward, the absence is
# *recoded* as the explicit category "Unknown". The downstream groupby
# will then produce an "Unknown" group, making the volume of incomplete
# records visible rather than hidden behind a NaN.
encounters["ward"] = encounters["ward"].fillna("Unknown")

# --- Step 4: transform -----------------------------------------
# The margin requires the reimbursement, which lives in the DRG catalogue.
# It is therefore computed after the merge in Step 6.

# --- Step 5: aggregate (preview, before merge) -----------------
# Average length of stay per ward — answers the first question.
los_by_ward = (encounters
               .groupby("ward")["los_days"]
               .agg(["mean", "median", "count"])
               .round(2)
               .sort_values("mean", ascending=False))
print("\nLength of stay by ward:\n", los_by_ward)

# --- Step 6: merge ---------------------------------------------
# Two consecutive *left joins* enrich the encounter table with information
# from the DRG catalogue and the patient register.
#   how="left"      retains every row of the left-hand table (the encounters)
#                   and adds the matching columns from the right-hand table;
#                   rows of the right table without a matching key are not
#                   added (no row is duplicated, no row is lost on the left).
#   validate="m:1"  asserts the expected *cardinality* of the join — that
#                   is, how many rows on one side correspond to how many on
#                   the other. "m:1" reads as "many-to-one": many encounters
#                   share one DRG catalogue entry; many encounters share one
#                   patient record. If the assumption is violated (for
#                   example a duplicate patient_id in 'patients'), pandas
#                   raises an exception immediately, catching errors that
#                   would otherwise silently inflate the result by joining
#                   each encounter to multiple rows.
enriched = (encounters
            .merge(drg_catalog, on="drg_code",   how="left", validate="m:1")
            .merge(patients,    on="patient_id", how="left", validate="m:1"))

# Column arithmetic in pandas is *vectorised*: the subtraction is applied
# to every row in a single operation dispatched to compiled code, not by
# a Python loop. This is markedly faster and more concise than iterating.
# The comparison ' < 0 ' similarly produces a boolean Series — True where
# the margin is negative, False otherwise.
enriched["margin_eur"] = enriched["reimbursement_eur"] - enriched["cost_eur"]
enriched["is_loss"]    = enriched["margin_eur"] < 0

margin_by_drg = (enriched
                 .groupby("drg_code")
                 .agg(total_margin=("margin_eur", "sum"),
                      n_cases    =("encounter_id", "count"))
                 .round(2)
                 .sort_values("total_margin"))
print("\nMargin by DRG:\n", margin_by_drg)

# --- Step 7: visualise -----------------------------------------
fig, axes = plt.subplots(1, 2, figsize=(12, 5))

axes[0].bar(los_by_ward.index, los_by_ward["mean"], color="#1565C0")
axes[0].set_title("Average length of stay by ward")
axes[0].set_xlabel("Ward")
axes[0].set_ylabel("Mean length of stay (days)")
axes[0].tick_params(axis="x", rotation=20)

# A *list comprehension* produces one colour per DRG: red where the total
# margin is negative, green where it is positive. Visual encoding of the
# sign communicates the result more directly than the bar length alone.
colors = ["#C62828" if v < 0 else "#2E7D32"
          for v in margin_by_drg["total_margin"]]
axes[1].barh(margin_by_drg.index, margin_by_drg["total_margin"], color=colors)
# axvline draws a vertical reference line on the second axes; here at x=0,
# marking the boundary between loss-making and profitable DRGs.
axes[1].axvline(0, color="black", linewidth=0.8)
axes[1].set_title("Total margin by DRG")
axes[1].set_xlabel("Margin (EUR)")

# tight_layout reconciles overlapping labels by recomputing the spacing
# between axes. savefig then writes the figure to disk:
#   dpi=200             sets a high raster resolution (dots per inch);
#   bbox_inches="tight" crops away surrounding whitespace, ensuring that
#                       rotated tick labels and titles are not cut off.
# savefig must be called *before* plt.show(), since plt.show may clear
# the figure on some backends, leaving savefig to write an empty canvas.
plt.tight_layout()
plt.savefig("saint_marys_summary.png", dpi=200, bbox_inches="tight")
plt.show()

# --- Step 8: export --------------------------------------------
Path("output").mkdir(exist_ok=True)
los_by_ward.to_csv("output/los_by_ward.csv")
margin_by_drg.to_csv("output/margin_by_drg.csv")
enriched.to_csv("output/encounters_enriched.csv", index=False)

print("\nDone. Summaries and the enriched table are in ./output/")

The synthetic data is regenerated with a fixed random seed, so the numerical results are reproducible. To work with the actual Saint Mary's files, replace Step 1 with pd.read_csv(...) calls against data/patients.csv, data/encounters.csv, and data/drg_catalog.csv.
